**Core selected features**

- `US-1` and `US-2` were selected because the assignment explicitly calls for use cases and the project already had a workable Iteration 2 chat shell that could support accessibility enhancements without a full redesign.
- `US-3` was selected because it directly satisfies the customer need and is the most distinctive feature of the final project.

- `US-4` was kept because secure password handling is already present in the finished app, strengthens the quality of the submission, and aligns with good software-engineering practice.

### 3.2 UI Design Goals

The individual iteration intentionally keeps the visual language of the group project instead of replacing it with a completely different UI. This was important for two reasons:

- the assignment asks for individual additions on top of the project rather than a disconnected rebuild
- the existing group HTML already had a recognizable structure that matched the demo flow

The preserved design language includes:

- centered white cards for login, sign-up, forgot-password, and reset-password
- three framed persona cards for character selection
- a two-column chat layout with history on the left and the active conversation on the right
- a red logout button and minimal blue/green action buttons

The only major UI additions are:

- a compact accessibility toolbar in the chat workspace
- three side-by-side response cards instead of a single AI reply block
- clearer starred-conversation rendering in the history sidebar

Several smaller UI details are also important to document accurately:

- the chat header shows the logged-in email and the currently active persona chip
- the workspace includes a `Change persona` link, so persona switching is part of the main flow
- the history sidebar shows a dedicated `STARRED` section for pinned conversations
- the forgot-password page exposes a local development preview reset link instead of requiring email integration

This project followed a lightweight Agile approach appropriate for an individual iteration:

1. inspect the current Group Iteration 1 and Iteration 2 baseline

For **new or modified behavior during the individual iteration**, the workflow was:

1. define the smallest expected behavior
2. write or update Jasmine tests for that behavior
3. make the unit tests pass
4. integrate the behavior into the page or route
5. add or update a Cucumber.js scenario
6. run the scenario in Puppeteer
7. fix regressions before moving to the next slice

Because this work builds on an inherited group baseline, not every preserved legacy feature began life in a pure red-green-refactor sequence. To stay accurate, the best summary is:

- **new individual features and bug fixes were handled with a unit-first workflow**
- **inherited features were stabilized and then brought under stronger regression coverage before further refinement**

## 7. Testing Strategy, Test Suites, And Traceability

### 7.1 Testing Philosophy

The testing strategy uses three layers:

1. **page-level Jasmine tests** for browser-page logic and DOM behavior
2. **logic-level Jasmine tests** for storage, services, security, and data behavior
3. **Cucumber.js plus Puppeteer acceptance tests** for end-to-end user behavior in a real browser

This layered approach matters because no single test type is enough by itself:

- unit tests are faster and more precise for logic
- browser tests prove the real user workflow works
- acceptance tests are derived from user stories, not internal implementation details

### 7.8 Why Cucumber.js Is Sensitive To Wording

Cucumber.js connects English-like steps to JavaScript step definitions. For example, the sentence:

```gherkin
When I send the prompt "Explain automated testing"
```

must match a step definition pattern in `features/individual/step_definitions/appSteps.js`, such as:

```js
When('I send the prompt {string}', async function (prompt) {
  ...
});
```

If the sentence changes too much, Cucumber.js no longer knows which step definition to execute. That is why acceptance tests can feel "sensitive" to wording even when the app itself still works.

### 7.9 Automated Browser Testing With Puppeteer

Puppeteer is used as the browser driver for all Cucumber acceptance suites. Important implementation details:

- the stable grading path is headless Puppeteer through `npm run test:acceptance`
- a visible-browser mode is available through `npm run test:browser` or `npm run test:acceptance:visible`
- visible mode uses a small `slowMo` delay so the browser actions can be watched
- Google Chrome is preferred when available
- if Chrome is unavailable, Puppeteer falls back to bundled Chromium
- test runs use isolated `.test-data` storage so browser tests do not contaminate normal data

## 8. Software Architecture And Implementation

### 8.1 High-Level Architecture

Triad.AI is a Node.js + Express web application with static frontend pages and JSON-file persistence.

| Layer | Responsibility | Main Files |
| :--- | :--- | :--- |
| Presentation layer | HTML, CSS, and client-side JavaScript | `public/` |
| Routing layer | page serving and REST endpoints | `routes/pageRoutes.js`, `routes/authRoutes.js`, `routes/chatRoutes.js` |
| Application/service layer | validation, auth storage, preferences, conversations, model-response generation, Ollama helpers | `utils/` |
| Persistence layer | JSON files for users, sessions, reset tokens, preferences, and conversations | `data/` |
| Test layer | Jasmine specs, Cucumber features, Puppeteer world/support files | `spec/`, `features/individual/` |

At startup, the server ensures the data directory exists before handling requests. When the app is run directly through `npm start`, it also attempts an Ollama warm-up call in the background so the first live response can return faster.

### 8.2 Main Runtime Flow

The main prompt-to-response flow is:

1. a user logs in and receives a session cookie
2. the user selects a persona
3. the chat page loads the user's session, preferences, conversation history, and Ollama status
4. the user sends a prompt
5. the prompt is stored in the current conversation
6. the server generates three provider-specific responses
7. the grouped response is stored as a `model-group` message
8. the client renders `GPT`, `Gemini`, and `Claude` cards side by side

### 8.2.1 Persona And Provider Orchestration

The chat-generation path combines two dimensions of behavior:

- **persona instructions**, which determine tone
- **provider instructions**, which determine card style

Persona instructions currently map to:

- `Miss Sweetheart` -> warm, encouraging, empathetic
- `Mr. Professional` -> concise, practical, professional
- `Lord Silly The Ninth` -> playful, whimsical, slightly absurd

Provider instructions currently map to:

- `GPT` -> clear structure and concise organization
- `Gemini` -> analytical comparison and tradeoff framing
- `Claude` -> supportive explanation plus practical next steps

This means the final response a user sees is shaped by both the selected persona and the selected provider card. That is why the project can truthfully claim that personas are integrated into the multi-LLM comparison experience.

### 8.2.2 Conversation And Rendering Behavior

Several implementation details affect what the user sees:

- conversations are stored with `persona`, `title`, `pinned`, `updatedAt`, and `messages`
- new conversation titles are derived automatically from the first user prompt
- grouped model responses are stored as a single `model-group` message containing three provider responses
- the history sidebar preview shows either the latest user message or summarized model-group text
- when the current chat is empty, the interface shows an instructional empty state rather than an empty message area
- before final cards appear, the workspace shows a persona-specific "is thinking" bubble and then animates the card text into view

### 8.2.3 Server-Backed Mode And Local/Test Mode

The client supports two runtime modes:

- the normal **server-backed mode**, used when the app is running through Express
- a **local/demo mode**, used when no real HTTP server API is available, which helps page-level tests run in JSDOM and preserves basic UI behavior

In server-backed mode:

- sessions, preferences, and conversations are loaded from REST endpoints
- Ollama status is checked through the backend
- input is disabled when Ollama is offline

In local/demo mode:

- conversations and preferences use `localStorage`
- generated responses come from local demo text rather than live backend calls

This dual-mode design is part of why the page-level Jasmine suite can test realistic chat behavior without requiring a live browser server for every page test.

### 8.5 Database Design

The project uses JSON files instead of a relational database. For a class project of this size, that makes the data model easy to inspect and easy for TAs to reproduce locally.

| File | Record Shape | Purpose |
| :--- | :--- | :--- |
| `users.json` | `{ email, passwordHash, passwordSalt, createdAt, passwordUpdatedAt? }` | registered users with hashed passwords |
| `sessions.json` | `{ sessionId: { email, createdAt } }` | active server-side sessions |
| `resetTokens.json` | `{ token: { email, createdAt, expiresAt } }` | temporary reset tokens |

### 8.6 Security-Relevant Design Choices

Security is not only a feature claim in this project; it is also part of the design:

- passwords are hashed with `crypto.scryptSync`
- password verification uses `crypto.timingSafeEqual`
- each password gets its own salt
- session state is stored server-side and referenced by cookie
- reset tokens expire automatically
- protected chat routes require an authenticated session
