## 1. Requirements

### Customer Need

The customer wants an LLM application that remains accessible to visually impaired users, lets users compare multiple model outputs clearly, and preserves the Iteration 2 workflow that already exists around personas, conversation history, authentication, and Ollama integration.

### User Story 1

As a visually impaired user, I want to increase and decrease text size so that prompts and responses are easier to read.

Estimated points: 3

SMART check:

- Specific: adjusts chat prompt and response text size through dedicated controls
- Measurable: success is visible through a percentage indicator and larger rendered text
- Achievable: implemented as a compact toolbar control in the existing chat page
- Relevant: directly supports visually impaired users
- Timeboxed: completed within the Individual Iteration-1 delivery window

### User Story 2

As a visually impaired user, I want to enable bold text and high-contrast mode so that response text is easier to distinguish.

Estimated points: 3

SMART check:

- Specific: enables bold text and a high-contrast color scheme on the chat workspace
- Measurable: success is visible through the toggles, bold font weight, and contrasting colors
- Achievable: implemented without replacing the Iteration 2 page structure
- Relevant: improves readability and distinguishability for the target user
- Timeboxed: completed within the Individual Iteration-1 delivery window

### User Story 3

As a user, I want to see multiple LLM responses in separate labeled sections so that I can compare outputs clearly.

Estimated points: 5

SMART check:

- Specific: shows GPT, Gemini, and Claude responses in separate labeled cards
- Measurable: each prompt creates three clearly labeled response sections
- Achievable: implemented on top of the existing Ollama-backed chat flow
- Relevant: supports the main product purpose of comparing model outputs
- Timeboxed: completed within the Individual Iteration-1 delivery window

### Use Case And Scenario Count

- Solo-iteration use cases implemented: 3
- Acceptance-test scenarios for those use cases: 4
  - text size increase/decrease
  - bold text and high-contrast mode
  - three labeled response cards
  - persona switching validation for the active workspace
- Additional browser scenarios covering preserved project features: 4
  - account signup, login, persona entry, and logout
  - Ollama availability, warm-up, and animated response behavior
  - password recovery
  - conversation history management and conversation continuation

### Feature Selection Summary

These three features were selected because together they improve accessibility and model comparison while staying realistic to implement within one iteration and preserving the existing Group Iteration 2 architecture.

## 2. UI Design

### Page Count

The finished project has 6 pages:

1. Login page
2. Sign-up page
3. Forgot-password page
4. Reset-password page
5. Persona-selection page
6. Chat workspace page

### Page Transitions

1. A new user opens `/signup`, creates an account, and returns to `/`.
2. An existing user opens `/`, logs in, and goes to `/choose-player`.
3. The user selects a persona and enters `/chat?persona=...`.
4. The user can return to `/choose-player` from the chat header and switch the active chat persona.
5. A user who forgot their password opens `/forgot-password`, uses the preview reset link, and completes the reset flow on `/reset-password`.

### Visual Design Choice

The UI intentionally stays close to the original Iteration 2 development HTML and CSS:

- centered white login and form containers
- simple blue, green, and red buttons
- the original persona-card layout
- the original left-sidebar chat shell

Only minimal solo-iteration additions were made:

- a compact accessibility toolbar above the chat area
- text-size increase and decrease buttons
- bold and high-contrast toggles
- three labeled response cards instead of a single AI response bubble

### Development Order: Unit Tests Then Acceptance Tests

The implementation process followed this order:

1. identify the smallest page or logic behavior for a feature
2. verify that behavior with Jasmine first
3. iterate until the unit and page suites pass
4. then verify the same user-facing workflow with Cucumber.js and Puppeteer
5. rerun the full test stack after integration

### Why Cucumber.js Feels Sensitive To Wording

Cucumber.js maps Gherkin sentences to JavaScript step definitions. Because the sentences are executable, a wording change can break the mapping.

Example:

```gherkin
When I send the prompt "Explain automated testing"
```

must match a step definition such as:

```js
When('I send the prompt {string}', async function (prompt) {
  ...
});
```

If the wording changes too much, Cucumber no longer knows which JavaScript function to run.

## 5. Software Architecture

### Routing Table

| Method | Route | Purpose |
| :--- | :--- | :--- |
| `GET` | `/` | Render login page |
| `GET` | `/signup` | Render sign-up page |
| `GET` | `/forgot-password` | Render forgot-password page |
| `GET` | `/reset-password` | Render reset-password page |
| `GET` | `/choose-player` | Render persona-selection page |
| `GET` | `/chat` | Render chat workspace |

### REST API Table

| Method | Route | Purpose |
| :--- | :--- | :--- |
| `POST` | `/api/signup` | Create a new account |
| `POST` | `/api/login` | Authenticate a user and create a session |
| `POST` | `/api/logout` | Remove the current session |
| `GET` | `/api/session` | Return session information |
| `POST` | `/api/forgot-password` | Generate a reset token and preview link |
| `POST` | `/api/reset-password` | Set a new password from a reset token |
| `GET` | `/api/preferences` | Load accessibility preferences |
| `PUT` | `/api/preferences` | Save accessibility preferences |
| `GET` | `/api/ollama-status` | Report whether Ollama is available |
| `GET` | `/api/warmup` | Trigger a warm-up call |
| `GET` | `/api/conversations` | Load saved conversation history |
| `POST` | `/api/conversations` | Create a new conversation |
| `PATCH` | `/api/conversations/:conversationId` | Update pin/title/persona metadata |
| `DELETE` | `/api/conversations/:conversationId` | Delete a conversation |
| `POST` | `/api/conversations/:conversationId/messages` | Save a user prompt |
| `POST` | `/api/conversations/:conversationId/ai-reply` | Generate and save three model responses |

### Database Design

The project uses file-based JSON storage:

| File | Purpose |
| :--- | :--- |
| `users.json` | user account records with hashed passwords |
| `sessions.json` | active login sessions |
| `resetTokens.json` | password-reset tokens and expiry times |
| `preferences.json` | accessibility settings |
| `conversations.json` | saved conversation history and model responses |

### Architecture Summary

1. `server.js` creates the Express app and starts the server.
2. `routes/pageRoutes.js` serves the page flow and protects chat and persona pages.
3. `routes/authRoutes.js` handles signup, login, logout, and password reset.
4. `routes/chatRoutes.js` handles preferences, Ollama state, conversation APIs, and persona updates.
5. `utils/` contains storage, authentication, conversation, and Ollama logic.
6. `public/` contains the Iteration 2 style frontend with minimal accessibility and comparison additions.
