### 6.4 How Acceptance Tests Were Derived From The Use Cases

The acceptance tests were not chosen arbitrarily. Each Gherkin scenario is a direct translation of a use case into visible browser behavior.

Example derivation for US-3:

1. user submits one prompt
2. system should generate multiple responses
3. user must be able to compare them clearly
4. therefore the scenario checks for three labeled sections with non-empty text

Example Gherkin excerpt:

```gherkin
Scenario: A prompt returns GPT, Gemini, and Claude-style answers
  Given I have a registered solo iteration account
  And I am logged in on the chat page with the "sweetheart" persona
  When I send the prompt "Give me three study tips for a software engineering class"
  Then I should see a response card for "GPT"
  And I should see a response card for "Gemini"
  And I should see a response card for "Claude"
```

Example derivation for the Iteration 2 history feature:

1. user creates conversations
2. user stars a conversation
3. user logs out and returns
4. the same starred conversation should still appear in the STARRED section

This became a browser scenario because persistence across sessions can only be trusted when exercised through the full application, not just a unit test.

### 6.5 Why Cucumber.js Is Sensitive To Wording

Cucumber.js executes natural-language steps by matching each sentence to a JavaScript step definition. For example:

```gherkin
When I send the prompt "Explain automated testing"
```

must match a step definition in `features/individual/step_definitions/appSteps.js` such as:

```js
When('I send the prompt {string}', async function (prompt) {
  ...
});
```

If the sentence changes too much, Cucumber.js no longer knows which JavaScript function should run. That is why seemingly small wording changes can make a test fail even when the browser behavior itself has not changed.

## 7. Software Architecture And Implementation

### 7.1 High-Level Architecture

The system is a small Node.js and Express web application with a static frontend and JSON-file persistence layer.

Architecture layers:

1. `public/` serves the client pages and page-specific JavaScript
2. `routes/` maps HTTP routes and REST endpoints to the correct handlers
3. `utils/` contains storage, validation, session, model-response, and Ollama helpers
4. `data/` stores JSON files for users, sessions, reset tokens, preferences, and conversations

### 7.2 Main Runtime Flow

1. the user signs up or logs in
2. the server creates a session cookie
3. the user selects a persona
4. the chat page loads preferences, session data, conversation history, and Ollama status
5. when the user sends a prompt, the prompt is stored in conversation history
6. the backend generates three provider-specific responses
7. the response group is stored and rendered as three labeled comparison cards

### 7.5 Database Design

The project uses JSON files rather than a relational database. This was appropriate for the size of the assignment and also keeps the data structures easy for the TAs to inspect.

| File | Record Structure | Purpose |
| :--- | :--- | :--- |
| `users.json` | `{ email, passwordHash, passwordSalt, createdAt, passwordUpdatedAt? }` | stores registered accounts without plaintext passwords |
| `sessions.json` | `{ sessionId: { email, createdAt } }` | stores active session metadata |
| `resetTokens.json` | `{ token: { email, createdAt, expiresAt } }` | stores short-lived password reset tokens |
| `preferences.json` | `{ email: { boldText, highContrast, fontScale } }` | stores accessibility preferences by user |
| `conversations.json` | `{ email: [conversation, ...] }` | stores conversation history by user |

Conversation record shape:

```json
{
  "id": "conv-...",
  "title": "Pinned study note",
  "persona": "sweetheart",
  "pinned": true,
  "updatedAt": 1710000000000,
  "messages": [
    {
      "type": "user-bubble",
      "text": "Pinned study note",
      "at": 1710000000001
    },
    {
      "type": "model-group",
      "responses": [
        {
          "providerId": "openai",
          "providerLabel": "GPT",
          "mode": "ollama-live",
          "text": "..."
        }
      ],
      "at": 1710000000200
    }
  ]
}
```

### 7.6 Security-Relevant Design Choices

- passwords are hashed with `crypto.scryptSync`
- sessions are stored server-side and referenced via cookies
- chat and persona routes are protected
- password reset requires a valid token with an expiry time
- browser tests verify the user must authenticate before reaching protected workflow pages
